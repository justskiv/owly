# Миграция стилей на Tailwind

> **Статус:** отложено. На старте проекта CSS перенесён из
> `design/tuzov-os-design-mock.html` как есть (голый CSS с
> CSS-переменными). Tailwind идёт в бэклог.

## Зачем

В `CLAUDE.md` проекта зафиксировано «Tailwind для стилей». На
стартовом этапе мы нарочно не пошли туда, потому что:

1. HTML-мок дизайнера — полностью готовая референс-реализация на
   голом CSS. Переносить 1-в-1 в React быстрее, чем
   переформулировать все классы в Tailwind.
2. Дизайн-спека построена на CSS-переменных (`--bg-*`, `--text-*`,
   `--work` и т.д.) — это естественно ложится на `:root`, не на
   `tailwind.config`.
3. Утилитарная природа Tailwind нам пока не нужна: дизайн-спека
   фиксированная, компоненты компонуются статично, «рерайт на
   лету» не требуется.

Но сидеть на голом CSS бесконечно — плохо:
- Тяжелее рефакторить
- Теряем типобезопасность Tailwind v4 + IDE-подсказки
- CLAUDE.md становится нерелевантным

## Когда мигрировать

После того, как все 6 фаз стабилизировались в коде и UI не меняется
каждый день. Ориентир — **после фазы 4 или фазы 5**. К этому моменту:
- Станут ясны реальные паттерны компоновки
- Будет видно, какие «компоненты» переиспользуются (блоки,
  бейджи, карточки, модалки) — их вынесем в `@apply`-классы или
  React-компоненты с Tailwind-классами
- Дизайн-токены стабилизируются → их можно зафиксировать в
  `tailwind.config.ts` через `theme.extend.colors =
  { 'bg-deep': 'var(--bg-deep)' }` и т.п.

## План миграции (когда дойдут руки)

1. Настроить Tailwind v4 + `@tailwindcss/vite`
2. Зафиксировать токены в `tailwind.config.ts`:
   - Цвета: `colors.bg.deep = 'var(--bg-deep)'` и т.д.
   - Шрифты: `fontFamily.sans = 'Outfit, ...'`,
     `fontFamily.mono = 'JetBrains Mono, ...'`
   - Радиусы: `borderRadius.xs/sm/md/lg/xl`
   - Font-size: шкала 2xs/xs/sm/md/lg/xl/2xl
   - Spacing: уже 4px-grid в Tailwind
3. Оставить CSS-переменные в `globals.css` как источник правды
   (Tailwind будет читать их через `var()`)
4. Постепенно заменять CSS-классы `.tb`, `.pi`, `.erow` и т.п. на
   набор Tailwind-классов или на React-компоненты с Tailwind
5. Сложные компоненты (heatmap, sparkline, bar-chart, pipeline)
   оставить на CSS/inline-стилях — там Tailwind не даёт выигрыша
6. Удалить вручную написанный CSS после миграции каждой секции
7. Обновить CLAUDE.md: «Tailwind — основной стек стилизации»

## Что можно делать уже сейчас

- Новый CSS (если что-то нужно добавить за рамками мока) писать
  Tailwind-классами, не добавляя в общий `globals.css`
- Исключение — если конкретная задача требует глобального CSS или
  сложной селекторной логики, которую не вытянуть через Tailwind
  (это стоит обсудить)
